Scope Management

bucSD015.htm / buc Draft Version 1.4 / 1997-11-29

 

Project Scope Management includes the processes required to ensure that the Y2K programme includes all the work required, and only the work required, to complete the programme successfully. It is primarily concerned with defining and controlling what is or is not included in the Y2K programme and what is under the scope of a specific Sub-Group and what is left without subvising to the organisations.

Overview

Initiation—committing the organisation to begin the next phase of the programme.

Scope Planning—developing a written scope statement as the basis for future programme decisions.

Scope Definition—subdividing the major programme deliverables into smaller, more manageable components (projects,Units,Products and Items).

Scope Verification—formalizing acceptance of the project scope.

Scope Change Control—Controlling changes to project scope

Scope Terminology

Project Scope
(PJ)		 the work that must be done by a Sub-Group in order to deliver compliance for assigned units.

Code = PJnnn ( nnn= Project number)
Unit
(U)		 Sign Off or Testing Unit
A bunch of items at a dedicated Organisation, forming a logical unit in addressing the smallest division for which tests are reasonable and for which the ProgMan has to sign for progressive compliance.

So called "Covered Units" are defined by the Organisation and/or PrOf with the goal to allocate identical units above several organisations for providing synergies.
For all remaining, so called "Non Covered Units" (which are not defined by the PrOf) the OrgMan's is not supported to have them compliant.
The OrgHead is responsible that all items within his organization are part of such well defined units.

Code = Uoooooooooo_uuu (oooooooooo=Organisation Code + uuu = Unit Code (where 001 -499 = covered, 500-999 = non covered ) )

Atom
(A)		 whereas an item must be understood as a Set of all installaed products at a dedicated organisation, an atom is its installation on a dedicated device
example: MS Access 97 running on a specific PC.

Atoms are beyond the scope of the PrOf but within the scope of Sub-groups
Item
(I)		 an item is everything which can be seen within a dedicated organisation. Sometimes an item is related exactly to one well defined Product. Responisbility and control upon an item is up to the organisation where it is located..

Code = Ioooooooooo_iiii ( oooooooooo=OrganisationCode & iiii = Item Code )

For the 1st level classification of items see the "Category" Attribute.
The Category classificat of Items and Products are the same
Product
(P)		 Is a product which can be defined precisely in terms of scope and function.
Responisbility for an item is a dedicates Sub-Group and is handled under the control of the PrOf.

The same product can be installed in various Organisations. Such an installation is then called "Item")

For the 1st level classification of Products see the "Category" Attribute
The Category classification of Item and Product are the same

Device a device is a well defined, singular piece of equipement that can be moved by hand:
examples:  Faxmachine at a dedicated location or organisation
Equipment Logical set of devices to serve a business process
Chunking The procedure for subdividing the major programme into smaller, more manageable entities.
Kernel A Kernel is a work unit that consist of those programs and data stores that will be changed simultaneously.

Cirteria for composition of a Kernel are:

  • Interaction of data between applications components
  • Application component inventory & relationships
  • ‘Intuitive’ information from application users and subject matter experts
  • Production schedules"

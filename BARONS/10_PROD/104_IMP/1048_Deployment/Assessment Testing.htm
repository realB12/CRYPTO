Testing Levels for Assessment Test

Assessment Test Definition

Assessment Test is done mainly during the Y2K projects Assessment phase to verify in a pragmatic and quick approach that no major conversion is needed. Normally it is performed by the developer.

Assessment Testing splits up in static component analysis and dynamic testing levels

Dynamic Testing Levels

Components

An Assessment Component is an idependant piece of code not containing external calles nor includes of other components. A component normally is used as an include or as an external function by modules and units. Sometimes a component is written in another language (ASSEMBLER) or bought from a vendor.

  • (Date) routines
  • Sorting Routines
  • Data Access modules (SQL Statements, Called Procedures)
  • Vendor code
  • Vendor application skeletons (4GL Components)

So Component testing is testing a single Component in isolation. Unit testing verifies that the unit works correctly based on some predefined minimal set of requirements.

A change in a Component may heavily trigger recompilation of dependent modules and units

Modules

A Module is piece of code repeatetly called or included by other modules or units. A Modul can contain external calles or includes other moduls or components

  • Called Functions, Procedures, SubRoutines
  • Screen Handling / Verification procedures

A change in a Component may heavily trigger recompilation of dependent modules and units

Unit

Compilation Unit, normally a program with a user interface or batch job designed to handle a certain activity

Sequence

Testing a collection of Units belonging to the same application including services and non "testable" or not tested entieties to check logical behaviour and user acceptance for the appication. This test may be repeated by the user as part of the Validation Tests. Often a seqence test shows whether Interfaces have to be tested.

Internal Interface

Testing the dataexchange between  Units of the same application  via datasources or buffers which belong exclusivly to this application

Simulating interfaces and linkages with other items and/or units to validate that information is being passed correctly. This is useful if additional interfaces will be added to the system during the conversion process or if problems between two or more modules cannot be isolated
The purpose is to find out whether the interfaces data is written correctly to the buffer and read again correctly from the other side
Interface testing normally is performed by the developers

External Interface

Testing that writing/reading data to the interface of an external application is correct.

System,

testing a unit or component on the final, compliant system (software,   utilities, network and hardware). It is used as a baseline for performance, load and stress testing.
System  Testing normally is done by a team consisting of application specialist, technical experts and  usesr.

Field

Testing a collection of Units that constitute a complete business activity. This includes foreign units (belonging to a different application or system), services and non "testable" or not tested entieties within the same application.
Activity test will prove logical correctnes and user acceptance within your organisaiton. This test may be repeated by the user as part of the Validation Tests.

Group

Cluster testing over WAN within the KUONI Group

Street

Coordination of this kind of testing will be done be the Programme Office

Applications are frequently connected with other applications to provide a higher or deeper level of functionality. Data may be shared between applications or systems. Multiple applications or systems may be involved in such an environment. This is the typical environment for Year2000 projects. Intersystem testing is required to confirm that the connection functions properly between the applications. This test determines that the proper parameters and data are correctly passed between applications, and proper coordination and timing of each function exists between applications.

CONDITIONAL COMPLIANCE TESTING

see bucSD092.pdf page 17

The ideal environment for compliance testing is to have a complete platform i. e. hardware, operating system software and application software available at the same time. The reality is that all the environment components may not be available until too late in the process to provide adequate time to perform compliance testing. When this situation occurs, the concept of conditional compliance testing should be considered. Conditional compliance testing implies that a logical system can be tested and certified in an environment that is not yet certified as long as the interface to the non-compliant component can be controlled in such a manner as to provide the necessary date information. The diagram below depicts the conditional compliance testing for a logical system utilizing a date application programming interface (API) in an environment where the operating platform is not yet compliant.

When testing of the logical system is completed, it can be certified conditionally on the compliance testing of the operating platform and the date API. When the platform is complete and has been certified, the date API can be tested and certified. When the date API is certified, the logical system can be considered fully tested for Year 2000 compliance.

Utilizing this approach, compliance testing of logical systems and operating platforms can be done in parallel.

Abstract

During Detailed Item Analysis, Item Environment Analysis is the first step to be undertaken..

The goal of this process is to understand how the item is linked to its environment.

By drawing the item's Environment Diagram we show all supporting items in isolation.

The item's Interface Diagrams show the dataflow from/to the item in a detail, so that we are sure that nothing is forgotten what is relevant to make the interface running..

Both diagrams will support all following activities like detailed assessment, test planning and testing and both diagrams are created by (re)arranging all supporting/interfacing  itmes on the itemmap (by resolving all locigal units) so that the new drawing is from the item's point of view.

The results of these process must be part of the detailed item analysis report.

Process Overview

Input Output Description
Assess the item exising documentation, userguides, handbooks, printouts fo online help etc.. Item Analysis Report Describe here at least in short what the purpose of the item is or add reference to describing documents (user guides, programming guidelines etc.).

The goal here is to summarize in few words the scope and function of the item from a users point of view so that the scope of the following steps are clear and we are sure what belongs to the item and where its boundaries are (expecially this supports further testplanning).

Perhaps you add her some brainstormed info and keywords describing the items issues concerning Y2K.

Compile the Items Environment Matrix

Have a list with at least the ItemNumber, the actual compliant status and the person in charge to deliver compliance of the supporting items and the supporting items of interfaces . Since the owner of the item is in charge to make sure that all supporting items (direct and those of  interfaces) are compliant as well (and when not he is responisble to rise the issue to the Programme Office) the following list will be used to check the status of these.
